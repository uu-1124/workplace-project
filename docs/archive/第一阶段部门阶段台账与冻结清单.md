| 销售商机、报价、活动回写链路 | `sales` | 冻结兼容 | 属于后续商业化闭环，不是第一阶段内部智能体平台内核验证 | 否 | 有销售阶段 PRD、数据模型和 smoke 后恢复 |
| 市场增长与获客投放链路 | `growth` | 冻结兼容 | 增长最容易把第一阶段扩成多页面、多指标演示，当前不作为内核主线 | 否 | 有增长阶段目标、渠道策略、指标定义后恢复 |
| `workspace.ts` 中旧工作台常量 | 多部门 | 冻结参考 | 可作为历史兼容或旧页面数据源，但不作为第一阶段新增功能主契约 | 否 | 只有当前端主入口重新消费它们时，才按实际调用面清理 |
| candidate 目录 | 多部门 | 暂停作为主线 | candidate 只能对照取样，不能替换主项目继续开发 | 否 | 经过明确合并决策和差异审查后才能迁移 |
| 完整上市 / 销售单线预览模式 | `sales` / 多部门 | 兼容模式 | 老板默认入口已收口到第一阶段主链，兼容模式只保留历史演练能力 | 否 | 后续阶段把兼容模式重新定义为正式阶段入口 |

## 4. 继续开发检查口径

每次继续开发前，先回答四个问题：

1. 这次改动是否只服务 `product -> operations -> risk`？
2. 是否会让 `sales / growth` 重新影响默认入口、默认指标或默认验收？
3. 是否会把执行结果包和部门标准交付物混为一谈？
4. 是否能用第一阶段后端契约测试、前端构建、第一阶段 UI smoke 验证？

只要第 1 题答案是否，或者第 2、3 题答案是，就不应直接写代码，应先拆出后续阶段任务。

## 5. 当前推荐验收命令

后续每补一个第一阶段闭环缺口，至少跑：

```bash
pytest -q tests/test_dispatch_command_state_contract.py -k "first_phase_workspace"
npm run build
npm run test:phase1-ui
```

如果改动触碰销售兼容链路，再额外跑销售相关回归，但销售回归不替代第一阶段默认验收。
