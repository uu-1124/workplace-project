# 第一阶段扫描结论与继续开发建议

日期：2026-07-06

## 1. 结论先说

当前项目**已经可以继续写代码**，不需要再等新计划书。

但继续开发只能按下面 3 条执行：

1. 只以 `workplace-api` 和 `workplace-desktop-electron` 为主项目继续推进，不切到 candidate 目录。
2. 只推进第一阶段主链：`product -> operations -> risk`。
3. 一次只补一个闭环缺口，不再并行扩部门、扩执行器、扩页面体系。

如果不按这 3 条收口，后面最容易再次回到“代码很多，但主链不稳”的状态。

## 2. 四份计划书写得怎么样

### 2.1 `企业级部门热插拔实施指南（最终蓝图）`

评价：**方向清楚，适合做长期北极星，不适合直接派当前开发任务。**

优点：

- 把平台和外部软件的边界讲清了。
- 把最终部门形态讲清了。
- 明确第一阶段不是最终形态。

问题：

- 它解决的是“最终要成为什么”，不是“本周该改什么”。
- 如果直接拿它驱动编码，容易把销售、增长、企业化能力一起带进来。

### 2.2 `内部实现分层设计.md`

评价：**四份里工程价值最高。**

优点：

- 准确抓到了当前项目的真实风险不是代码少，而是顺序乱。
- 把“前端骨架 -> 对象 -> 状态 -> 服务 -> API -> ViewModel -> 页面 -> 测试 -> 文档”的施工顺序钉住了。
- 把“跨阶段偷跑”定义成明确禁止项。

问题：

- 这是施工纪律文档，不是当前项目进度台账。
- 它告诉你“怎么做”，但不告诉你“现在哪些部门已经超前，哪些还没到位”。

### 2.3 `第一阶段部门执行顺序计划书.md`

评价：**当前最可直接执行的一份。**

优点：

- 第一阶段唯一目标非常明确。
- 主链部门、冻结部门、明确不做范围都写清了。
- 老板主界面、员工工作台、一期部门工作台的职责边界清楚。

问题：

- 它定义了第一阶段该怎么打，但没有把“当前各部门已开发到哪”做成现状表。
- 所以它能指导编码方向，但不能单独解决你现在最头疼的“各部门进度不一”。

### 2.4 `第一阶段工程规格.md`

评价：**写得及时，而且现在已经是继续开发的直接契约。**

优点：

- 把第一阶段真正绑定到了现有代码对象和接口上。
- 明确主项目目录才是默认继续开发位置。
- 明确销售、增长不要删，但必须收口。

问题：

- 这是工程契约，不是排期文档。
- 它解决“怎么不写乱”，不解决“谁先停、谁先做、谁先冻结”的项目管理问题。

## 4. 什么时候可以开始继续开发

答案：**现在就可以。**

不是“等文档补完再说”，而是：

> 文档已经够了，接下来缺的不是新计划书，而是严格按计划书执行。

当前继续开发的正确方式不是再补一轮大规划，而是按下面顺序推进：

1. 只找第一阶段主链上的真实缺口。
2. 一次补一个缺口。
3. 每补一个缺口就用第一阶段测试和构建验证。
4. 不把销售、增长、企业化能力混进默认发布口径。

## 5. 你现在最头疼的问题：各部门进度不一，怎么办

这是当前最关键的管理问题。

我的判断是：

> **不要试图把所有部门硬拉回同一进度。**

应该分三类处理。

### A 类：未达到第一阶段的部门

例如还没把部门工作台、对象、状态、交接链接上第一阶段主链的部门。

处理方式：

- 继续补到第一阶段最低可运行标准。
- 只补主链必需能力。
- 不补未来阶段深度能力。

### B 类：刚好处于第一阶段的部门

例如 `product / operations / risk` 这三个当前主链部门。

处理方式：

- 继续作为唯一活跃开发线。
- 所有默认入口、默认测试、默认发布口径都围绕它们。
- 新工作只允许证明主链更稳，不允许顺手扩别的阶段。

### C 类：已经明显超前第一阶段的部门

典型就是 `sales / growth` 这种代码规模已经不小、但又不该继续决定当前产品结构的部门。

处理方式：

- **不删除**
- **不回退**
- **不继续扩主线**
- **只做冻结兼容**

“冻结兼容”具体意思只有 4 句：

1. 页面还能开，避免现有演示和回归直接坏掉。
2. 接口还能读，避免牵连当前系统结构。
3. 不再让它们决定老板主界面、第一阶段 payload、默认测试门槛。
4. 后续只在明确进入下一阶段时再恢复主动开发。

这就是你现在最该做的，不是继续生成更多文档。

- 新计划书
- 新候选目录
- 新页面大改稿
- 新对象体系
- 新部门路线

只保留下面两种文档资产就够了：

1. **阶段规则文档**
   - 你现在已经有了，够用了。

2. **现状台账文档**
   - 你现在还缺这个。

也就是说，后面不是继续“生文档”，而是把当前项目现状压成一张能管理的表。

## 7. 目前最需要你排版/整理的内容

计划书里大方向大多已经写好了。

**没写完整的，不是原则，而是“当前现状映射表”。**

当前现状映射表已补在：`第一阶段部门阶段台账与冻结清单.md`。

如果你要补排版，我建议只补下面两张表，别再扩。

### 7.1 部门阶段台账

每个部门一行，字段建议固定为：

- 部门
- 当前开发阶段
- 是否属于第一阶段默认主链
- 当前默认入口
- 当前是否冻结
- 允许继续做什么
- 明确不再做什么
- 下一次解冻条件

建议先填这 5 个部门：

- product
- operations
- risk
- sales
- growth

### 7.2 第一阶段冻结清单

每项一行，字段建议固定为：

- 模块/页面/接口
- 所属部门
- 当前状态：活跃 / 冻结兼容 / 暂停
- 原因
- 是否影响默认发布
- 后续恢复条件

这张表的作用是防止后面又把超前部门拉回默认主线。
