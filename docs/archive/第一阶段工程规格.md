# 第一阶段工程规格

## 1. 文档定位

本文档是第一阶段的工程落地规格，不替代另外三份文档。

三份既有文档继续保留各自职责：

- `企业级部门热插拔实施指南(最终蓝图).md`：定义长期产品蓝图、最终部门形态、热插拔模型和企业化边界。
- `内部实现分层设计.md`：定义总施工顺序、内部层次、阶段纪律和跨阶段禁止事项。
- `第一阶段部门执行顺序计划书.md`：定义第一阶段的产品目标、部门顺序、模块顺序和验收口径。

本文档只回答一个问题：

> 按当前项目已经写出来的代码，第一阶段继续写代码时，哪些对象、接口、状态、前端数据结构和边界必须遵守。

因此本文档不是重新设计，也不是把三份文档压缩成一份。它是给后续编码 Agent 使用的工程契约。

## 2. 当前代码基线

本规格以当前主目录代码为基线：

- 后端主项目：`workplace-api`
- 前端主项目：`workplace-desktop-electron`
- 候选目录：`workplace-api-phase1-candidate`、`workplace-desktop-phase1-candidate`

第一阶段继续开发时，默认参考主项目目录，不默认切换到 candidate 目录。candidate 目录只能作为对照和取样来源，不能在没有明确决策的情况下替换主项目。

当前已经存在的合理实现应保留：

- `business_commands` 已经是老板指令对象。
- `department_work_items` 已经是部门工作项对象。
- `ai_artifacts` 已经承担部门产物、AI 产物、风险审查材料的承载。
- `owner_decisions` 已经承担老板拍板。
- `human_questions` 已经承担澄清问题。
- `work_item_dependencies` 已经承担部门间依赖。
- `dispatch_handoffs` 已经承担交接包。
- `adapter_run_logs` 已经承担执行器运行记录。
- `member_department_permissions` 已经承担最小部门权限。
- `FirstPhaseWorkspacePayload` 已经是第一阶段前端主数据包。

当前不合理或超出第一阶段主链的实现不要删除，但要收口：

- 销售、增长相关代码已经有相当规模，第一阶段不继续扩张。
- `workspace.ts` 中大量销售、增长、旧工作台类型不作为第一阶段主契约。
- `firstPhaseWorkspaceMapper.ts` 中 `OWNER_DEPARTMENT_ORDER = ['growth', 'product', 'operations', 'risk', 'sales']` 不应被理解为第一阶段主链顺序。第一阶段主链只认 `product -> operations -> risk`，销售和增长只可作为冻结展示或未来阶段入口。
- `risk_review` 在第一阶段不强制新增独立表。当前应作为语义对象映射到 `ai_artifacts`、`owner_decisions`、`dispatch_handoffs` 和风险部门 `department_work_items`。

## 3. 第一阶段主链

第一阶段只完成一个可演示、可追踪、可审计的内部闭环：

```text
老板提出 business_command
-> 调度中枢预览并拆分 department_work_items
-> product 产出研发/交付材料
-> operations 接收交接并形成运营排期/执行计划
-> risk 审查产物、边界、风险
-> owner_decisions 触发老板拍板
-> 状态和 timeline 回流到老板主界面
```

第一阶段主链部门：

- `product`：研发工程部，负责把老板目标变成可执行工程产物。
- `operations`：交付运营部，负责接住研发产物并整理交付排期、执行计划、阻塞点。
- `risk`：财务法务风控部，负责审查边界、承诺、风险和是否允许进入下一步。

第一阶段冻结部门：

- `sales`：销售成交部。保留当前代码和页面，第一阶段不继续做销售闭环。
- `growth`：市场增长部。保留当前代码和页面，第一阶段不继续做增长投放闭环。

第一阶段不做：

- 完整 CRM。
- 完整增长投放系统。
- 完整企业权限中心。
- 多租户商业化硬壳。
- 任意外部工具的全自动危险操作。
- 为 `risk_review` 额外建表，除非现有映射无法表达验收所需信息。

## 4. 对象映射

后续编码时，优先复用当前对象，不轻易新增平行对象。

| 语义对象 | 当前后端表 | 当前前端类型 | 第一阶段要求 |
| --- | --- | --- | --- |
| 老板指令 | `business_commands` | `BusinessCommand` | 保留为主入口 |
| 部门工作项 | `department_work_items` | `DepartmentWorkItem` | 保留为部门任务主对象 |
| 部门产物 / AI 产物 | `ai_artifacts` | `AiArtifact` | 保留为产物和审查材料载体 |
| 老板拍板 | `owner_decisions` | `OwnerDecision` | 保留为老板决策入口 |
| 人工澄清 | `human_questions` | `HumanQuestion` | 保留为部门/老板问题入口 |
| 工作项依赖 | `work_item_dependencies` | `WorkItemDependency` | 保留为依赖关系 |
| 部门交接 | `dispatch_handoffs` | `DispatchHandoff` | 保留为 product 到 operations 等交接包 |
| 执行器运行 | `adapter_run_logs` | `AdapterRunLog` | 保留为 Claude/Codex/外部工具运行记录 |
| 部门权限 | `member_department_permissions` | `MemberDepartmentPermission` | 保留为第一阶段最小权限 |
| 风控审查 | 不单独建表 | `AiArtifact` + `OwnerDecision` | 作为语义对象实现，不新造表 |

### 4.1 `business_command`

必须表达：

- 谁创建：`created_by_member_id`
- 要做什么：`objective`
- 背景是什么：`context`
- 优先级：`priority`
- 当前状态：`status`
- 创建和更新时间：`created_at`、`updated_at`

允许状态：

- `draft`
- `dispatched`
- `in_progress`
- `waiting_decision`
- `completed`
- `cancelled`

第一阶段编码原则：

- 老板输入不能直接跳到执行器。
- 必须先经过预览、拆解或调度记录。
- `waiting_decision` 只能由待拍板事项、待审批产物、关键阻塞触发。

### 4.2 `department_work_item`

必须表达：

- 归属指令：`command_id`
- 归属部门：`department_id` / `department_key`
- 被分配人：`assigned_member_id`
- 标题、描述、期望产出：`title`、`description`、`expected_output`
- 状态：`status`
- 截止时间：`due_at`

允许状态：

- `todo`
- `working`
- `waiting_ai`
- `waiting_human_review`
- `waiting_owner_decision`
- `done`
- `blocked`

第一阶段编码原则：

- product、operations、risk 的页面都应以 `department_work_items` 为主列表来源。
- 页面上看起来像任务、阶段、审查项的东西，优先映射为 work item，不先新建局部对象。
- 只有明确是产物内容时才进入 `ai_artifacts`。

### 4.3 `ai_artifact`

必须表达：

- 归属指令：`command_id`
- 归属工作项：`work_item_id`
- 归属部门：`department_id` / `department_key`
- AI 或人工操作者：`produced_by_ai_worker_id`、`operated_by_member_id`
- 审核人：`reviewed_by_member_id`
- 标题、类型、正文：`title`、`artifact_type`、`content`
- 状态：`status`

允许状态：

- `draft`
- `waiting_human_review`
- `needs_revision`
- `ready_for_owner`
- `approved`
- `rejected`
- `archived`

第一阶段编码原则：

- product 的研发输出、operations 的交付计划、risk 的审查意见，都可以是 `ai_artifacts`。
- `artifact_type` 用于区分语义，例如 `department_output`、`risk_review`、`delivery_plan`、`code_change_plan`。
- 风控审查不必建独立表，先用 `artifact_type = risk_review` 或同类类型承载。

### 4.4 `owner_decision`

必须表达：

- 关联指令：`command_id`
- 关联产物：`artifact_id`
- 谁发起：`requested_by_member_id`
- 决策标题、提示、选项：`title`、`decision_prompt`、`options_json`
- 决策状态和结果：`status`、`decision`

允许状态：

- `pending`
- `approved`
- `rejected`
- `needs_more_info`
- `cancelled`

第一阶段编码原则：

- 预算、方向、对外承诺、最终通过，都必须进入 `owner_decisions`。
- 普通部门专业问题不要滥用老板拍板，应进入 `human_questions`。
- 老板拍板后必须能影响相关 artifact 或 work item 的状态。

### 4.5 `human_question`

必须表达：

- 关联指令：`command_id`
- 可选关联工作项：`work_item_id`
- 提问类型：`question_type`
- 问题正文：`question_text`
- 未回答影响：`impact_if_unanswered`
- 优先级和状态：`priority`、`status`

允许状态：

- `pending`
- `answered`
- `escalated`
- `cancelled`

第一阶段编码原则：

- 缺少事实时必须问，不允许 Agent 编造。
- 部门专业问题优先问对应部门负责人。
- 老板级问题才升级为 `owner_decisions`。

### 4.6 `dispatch_handoff`

必须表达：

- 关联指令：`command_id`
- 依赖关系：`dependency_id`
- 来源工作项和目标工作项：`source_work_item_id`、`target_work_item_id`
- 来源部门和目标部门：`from_department_*`、`to_department_*`
- 状态、说明、更新时间。

允许状态：

- `pending`
- `ready`
- `accepted`
- `returned`
- `completed`
- `blocked`
- `cancelled`

第一阶段编码原则：

- product 完成可交付材料后，operations 通过 handoff 接收。
- handoff 未 ready 时，目标部门不能假装已经有完整输入。
- return 必须带原因，并回流到上游。

### 4.7 `adapter_run_log`

必须表达：

- 归属指令：`command_id`
- 归属工作项：`work_item_id`
- 执行器：`adapter_id` / `adapter_key`
- 输入、输出、错误：`input_json`、`output_json`、`error_message`
- 状态：`status`
- 操作者和时间。

允许状态：

- `planned`
- `waiting_approval`
- `running`
- `succeeded`
- `failed`
- `cancelled`
- `skipped`

第一阶段编码原则：

- `adapter_run_logs` 是当前实现名，不要另起 `adapter_runs` 平行表。
- Claude Code 是第一优先执行器。
- Codex 可作为第二执行器，但第一阶段不要求多执行器智能调度。
- 外部执行器真实运行必须保留人工确认边界。

## 5. 状态机契约

### 5.1 Artifact 审核流转

当前后端已定义合法流转：

| 当前状态 | 可流转到 |
| --- | --- |
| `draft` | `needs_revision`, `ready_for_owner`, `approved`, `rejected`, `archived` |
| `waiting_human_review` | `needs_revision`, `ready_for_owner`, `approved`, `rejected`, `archived` |
| `needs_revision` | `needs_revision`, `ready_for_owner`, `approved`, `rejected`, `archived` |
| `ready_for_owner` | `approved`, `rejected`, `needs_revision`, `archived` |
| `approved` | 无 |
| `rejected` | 无 |
| `archived` | 无 |

编码要求：

- 前端按钮必须按这个状态机显示，不允许提供后端会拒绝的动作。
- `approved`、`rejected`、`archived` 视为终态。
- 若需要返工，不要直接从 `approved` 改回草稿，应新建修订工作项或新 artifact。

### 5.2 Adapter run 流转

当前后端允许：

| 当前状态 | action | 下一状态 |
| --- | --- | --- |
| `waiting_approval` | `approve` | `planned` |
| `waiting_approval` | `cancel` | `cancelled` |
| `waiting_approval` | `skip` | `skipped` |
| `planned` | `start` | `running` |
| `planned` | `succeed` | `succeeded` |
| `planned` | `fail` | `failed` |
| `planned` | `cancel` | `cancelled` |
| `planned` | `skip` | `skipped` |
| `running` | `succeed` | `succeeded` |
| `running` | `fail` | `failed` |
| `running` | `cancel` | `cancelled` |

编码要求：

- `waiting_approval` 必须由老板 approve 后才能进入 `planned`。
- 部门成员不能 approve 或 cancel 需要老板控制的执行器运行。
- `succeeded`、`failed`、`cancelled`、`skipped` 视为终态，不做隐式重开。重跑使用 replay 生成新记录或按当前 replay 接口处理。

### 5.3 Handoff 流转

第一阶段只要求以下动作稳定：

- `ready -> accepted`
- `ready -> returned`
- `accepted -> returned`
- `accepted -> completed`
- 阻塞时进入 `blocked`
- 放弃时进入 `cancelled`

编码要求：

- accept 要求 handoff 已经 `ready`。
- accept 后目标 work item 应进入 `working`。
- return 必须写明 reason。
- return 后上游页面必须能看到原因。

### 5.4 Work item 流转

第一阶段不需要复杂状态机，但必须遵守含义：

- `todo`：已创建，未开始。
- `working`：人工或部门正在处理。
- `waiting_ai`：正在等待执行器或 AI 产出。
- `waiting_human_review`：等待部门人工审核。
- `waiting_owner_decision`：等待老板拍板。
- `done`：完成。
- `blocked`：被事实缺失、依赖缺失、风险、执行失败阻塞。

编码要求：

- `blocked` 必须有 note、question、handoff return 或 event 解释。
- `done` 前必须存在可检查的 artifact、handoff 或明确人工说明。
- 页面不要只改本地状态，必须调用后端接口落库。

## 6. API 契约

第一阶段优先使用现有 dispatch 路由：

| 能力 | 方法和路径 | 用途 |
| --- | --- | --- |
| 指令预览 | `POST /api/dispatch/{team_id}/commands/preview` | 老板输入先生成预览、问题、拆解 |

| 创建指令 | `POST /api/dispatch/{team_id}/commands` | 创建 business command |

| 接收交接 | `POST /api/dispatch/{team_id}/handoffs/{handoff_id}/accept` | operations 等接收输入 |
| 退回交接 | `POST /api/dispatch/{team_id}/handoffs/{handoff_id}/return` | 输入不足时退回 |
| 回答问题 | `POST /api/dispatch/{team_id}/questions/{question_id}/answer` | 回答澄清问题 |
| 老板拍板 | `POST /api/dispatch/{team_id}/owner-decisions/{decision_id}/answer` | 回答 owner decision |
| 执行器动作 | `POST /api/dispatch/{team_id}/adapter-runs/{run_id}/action` | approve/start/succeed/fail/cancel/skip |
| 执行器重放 | `POST /api/dispatch/{team_id}/adapter-runs/{run_id}/replay` | 重新运行或复制运行上下文 |
| 执行器预览 | `POST /api/dispatch/{team_id}/adapter-runs/{run_id}/invoke-preview` | 预览外部调用 |
| 适配器包 | `POST /api/dispatch/{team_id}/adapter-runs/{run_id}/adapter-package` | 生成外部执行包 |
| 代码包 | `POST /api/dispatch/{team_id}/adapter-runs/{run_id}/code-package` | 生成 Claude/Codex 代码执行包 |
| 代码 Agent 执行 | `POST /api/dispatch/{team_id}/adapter-runs/{run_id}/code-agent-run` | 调用代码 Agent |
| 外部执行器执行 | `POST /api/dispatch/{team_id}/adapter-runs/{run_id}/external-run` | 调用外部适配器 |

编码要求：

- 前端新功能优先接这些接口，不绕过 `dispatch.py` 直接打零散业务接口。
- 如果接口返回的数据不够，优先扩展这些现有接口的 payload。
- 不为同一动作新增第二套 endpoint，除非现有路由语义明显不适合。

- `workspace.ts` 中旧对象可继续存在，但不作为第一阶段新增功能的主类型来源。
- 销售和增长页面不删除；第一阶段只做不破坏、不扩张。

## 8. 权限契约

当前后端权限规则可以支撑第一阶段：

- owner 可以管理所有工作项和执行器动作。
- 被分配成员可以管理自己的 work item。
- 拥有部门 `can_manage_tasks` 权限的成员可以管理该部门 work item。
- 部门权限包括：
  - `can_view`
  - `can_prompt_ai`
  - `can_review_artifacts`
  - `can_manage_tasks`
  - `can_request_owner_decision`
  - `can_manage_department_settings`

执行器权限：

- owner 可以执行所有 adapter run action。
- `approve`、`cancel` 只允许 owner。
- `start`、`succeed`、`fail`、`skip` 可由对应部门成员或具备 `can_manage_tasks` 的成员操作。

第一阶段不做完整 RBAC，但必须做到：

- 老板拍板不能被普通成员代替。
- 普通成员不能审批高风险执行器真实运行。
- 部门成员只能操作自己部门或授权部门的数据。
- 前端按钮要按权限隐藏或禁用，后端仍必须做最终校验。

## 9. 执行器契约

第一阶段执行器优先级：

1. Claude Code
2. Codex
3. 其他外部适配器只保留框架和手动确认路径

代码执行器必须接收的上下文：

- command：老板目标、背景、优先级。
- work item：部门任务、期望产出、当前状态。
- adapter run：执行器 id、输入 JSON、运行状态。
- department：部门 key、部门名、权限边界。
- known facts：老板已确认事实。
- artifact history：相关产物和审批结果。

代码执行器必须输出：

- summary：做了什么。
- changed_files：改了哪些文件。
- risks：还存在什么风险。
- next_steps：下一步建议。
- artifact_content：可回写到 `ai_artifacts` 的正文。
- status：成功、失败、需要人工处理。

真实执行边界：

- 默认 dry run。
- 真实运行必须有 `confirm_real_execution = true`。
- 危险操作必须要求明确确认文本。
- 执行失败必须写入 `adapter_run_logs.error_message`，不能只在前端 toast。

## 13. Smoke 验收链

第一阶段必须至少能跑通以下链路。

### 13.1 老板发起指令

输入：

- objective：一个真实业务目标。
- context：必要背景。
- priority：`high` 或 `urgent`。

验收：

- 能创建 `business_commands`。
- 能生成 product、operations、risk 相关 work items。
- 老板主界面能看到指令和主链进度。

### 13.2 Product 执行

动作：

- product 成员接收 work item。
- 触发 Claude Code 或生成执行器运行记录。
- 提交 `ai_artifacts`。
- 必要时请求 owner decision。

验收：

- work item 状态正确变化。
- adapter run 状态正确变化。
- artifact 可被审核。
- 老板能看到需要拍板的事项。

### 13.3 Operations 交接

动作：

- product 产物 ready 后生成 handoff。
- operations accept 或 return。
- return 必须写原因。

验收：

- handoff 状态正确。
- accept 后 operations work item 进入 `working`。
- return 后 product 或老板能看到原因。

### 13.4 Risk 审查

动作：

- risk 基于 artifact 或 work item 生成审查意见。
- 审查意见进入 `ai_artifacts`。
- 关键风险进入 `owner_decisions`。

验收：

- risk 页面能看到审查项。
- 老板能批准、拒绝或要求更多信息。
- 审批结果能回写 artifact 或 work item。

### 13.5 Timeline 和审计

动作：

- 创建指令、更新任务、提交产物、执行器运行、handoff、老板拍板。

验收：

- `GET /commands/{command_id}/state` 能返回主状态。
- 老板主界面 timeline 不为空。
- 状态解释能追溯到具体对象。
