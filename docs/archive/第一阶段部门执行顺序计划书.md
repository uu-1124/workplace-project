# 第一阶段部门执行顺序计划书

## 1. 文档目的

这份文档只回答一个最现实的问题：

> 接下来最近这一个阶段，到底先做什么、后做什么、哪些必须完成、哪些坚决不能碰。

它不是讲长期梦想的，也不是讲未来所有部门的。

它只服务最近的第一阶段作战。

这份文档承担 6 个职责：

1. 定义第一阶段唯一目标
2. 定义第一阶段范围和冻结区
3. 定义第一阶段前端优先顺序
4. 定义第一阶段模块顺序
5. 定义第一阶段 smoke 和里程碑
6. 定义第一阶段通过标准

---

## 2. 第一阶段唯一目标

第一阶段不追求“完整公司”。

第一阶段唯一目标只有一句话：

> 把 `老板目标 -> 调度中枢编排 -> 研发工程部 -> 交付运营部 -> 财务法务风控部 -> 调度复盘` 这条最小内部闭环跑通。

换句话说，第一阶段要证明的是：

1. 老板能下指令
2. 平台能拆任务
3. 部门能接任务
4. 外部软件能被真实调用
5. 结果能被回收
6. 审批能卡住并回流
7. 调度中枢能编排并回看完整链路

---

## 3. 第一阶段的核心判断

### 3.1 为什么第一阶段必须先定前端

因为如果前端骨架不稳定：

1. 老板没有稳定入口下命令和拍板
2. 员工没有稳定入口收任务和回写
3. 部门没有稳定入口承载业务对象
4. 后端接口会一直跟着页面漂移

所以第一阶段确实要先把前端方向定住。

### 3.2 但“前端先做”不等于“先把所有页面做完”

第一阶段前端要先做的是：

1. 一级界面信息架构
2. 老板主界面的最小指挥框架
3. 员工工作台
4. 一期部门页统一模板

第一阶段前端不先做的是：

1. 销售整套深度业务前台
2. 增长整套流水线
3. 大量视觉重绘
4. 把老板页做成传统总览大屏
5. 一堆未来阶段页面

---

## 4. 第一阶段范围

### 4.1 必做范围

第一阶段必须覆盖下面 5 类前端入口和 1 类执行器能力：

1. 老板主界面
2. 员工工作台
3. 研发工程工作台
4. 交付运营工作台
5. 财务法务风控工作台
6. 至少一个真 `CLI` 执行器

### 4.2 一期主链部门

第一阶段真正进入主链的业务域 / 部门只有：

1. `command`
2. `product`
3. `operations`
4. `risk`

这里要特别说明：

1. `command` 不是一个前端页面名
2. `command` 也不是普通业务部门
3. `command` 代表第一阶段的调度业务域，由调度中枢管理 `business_command`

### 4.3 保留但冻结

第一阶段保留但冻结：

1. `sales`
2. `growth`

“冻结”含义是：

1. 不删除代码
2. 不继续扩主线
3. 不让它们决定整体前端结构

### 4.4 明确不做

第一阶段不做：

1. 完整销售自动化
2. 完整增长自动化
3. `MCP`
4. Browser runner
5. GUI 自动化
6. 多租户
7. 企业级部署
8. 为了“更像企业级”而做大面积 UI 重绘

### 4.5 第一阶段账号与协同范围

第一阶段账号机制只做最小可运行版本，但必须真能用。

第一阶段必须支持：

1. 一个老板主账号
2. 老板主账号默认拥有全部门总权限，不需要手动给自己分配
3. 老板可创建员工子账号
4. 员工可被挂到一个或多个一期部门
5. 一人公司模式可直接成立，不要求必须先创建员工
6. 同部门可存在多名成员，但任务进入执行态时只能落到一个当前负责人

第一阶段明确不做：

1. SSO
2. MFA
3. 完整邀请激活流程
4. 复杂角色模板中心
5. 跨组织治理

## 5. 第一阶段前端产品定义

第一阶段前端只认下面这 3 类一级工作界面和 1 个系统界面。

### 5.1 老板主界面

老板主界面是第一主入口。

第一阶段它不是大总览屏，而是最终产品的最小可运行指挥框架。

它只负责下面 4 类内容：

1. 当前命令对话
   - 底部聊天输入框
   - 目标输入
   - 补充边界
   - 追问和改令

2. 当前命令处理过程
   - 系统正在拆分任务
   - 任务分配预览
   - 老板确认后开始下发

3. 当前命令执行反馈
   - 当前进度
   - 关键日志
   - 是否卡住

4. 右侧待处理 / 待老板决定操作台
   - 待拍板
   - 待补充
   - 待驳回 / 待确认

具体业务细节不在这里堆，统一下沉到部门工作台查看。

不允许继续往老板主界面里塞完整销售系统、未来增长系统和全局 BI 总览屏。

### 5.2 第一阶段前端与最终版前端的区别

第一阶段前端不是最终产品全貌，而是最终产品的最小可运行指挥框架。

第一阶段先做到：

1. 老板能通过聊天方式下命令、看拆分、做确认、看当前进度
2. 员工和部门都有稳定入口接任务、推进任务、回写结果
3. 主链能从命令发起一路跑到执行、交付、审批和复盘

最终版再继续补：

1. 多命令并行管理
2. 更完整的部门沉淀页和历史检索能力
3. 更成熟的跨部门协作、统计、沉淀和长期资产层
4. 更完整的企业化能力，而不是第一阶段就把所有东西堆满

### 5.3 员工工作台

员工工作台只负责：

1. 我的待办
2. 我的补充问题
3. 我的依赖和阻塞
4. 我的成果提交
5. 我的状态推进

它不是部门业务前台。

### 5.4 一期部门工作台

一期部门工作台只做 3 个：

1. 研发工程工作台
2. 交付运营工作台
3. 财务法务风控工作台

这三页必须统一成同一套模板结构：

1. `Brief`
2. `Pipeline`
3. `Execution`
4. `Artifacts`
5. `Review`

### 5.5 系统设置

设置页只负责：

1. 模型配置
2. 后端地址
3. 团队参数
4. 运行状态

它不属于主业务闭环。

### 5.6 第一阶段部门协同规则

第一阶段多人协同不走“所有人一起改一个对象”的路子。

固定按下面规则执行：

1. 老板主账号天然全权
2. 每个部门默认只认一个主负责人
3. 一个部门可以有多个执行成员
4. 任务先进入部门公共队列
5. 部门再把任务分到单一负责人
6. 做完后进入审核 / 交接 / 回流
7. 老板始终保留接管和重新分派权

第一阶段暂时不追求完整的多人实时协同编辑，只追求“队列清楚、责任清楚、回流清楚”。

### 5.7 第一阶段术语边界

第一阶段必须把下面三个词分开，不允许混用：

1. `老板主界面`
   - 前端入口
   - 负责老板输入命令、确认分派、查看当前命令状态
2. `调度中枢`
   - 系统内部编排内核
   - 负责拆解、分派、协同、阶段控制、审批挂点
3. `business_command`
   - 调度中枢管理的核心业务对象

调度中枢不是页面，不是老板工作台，也不是普通部门页。

### 5.8 第一阶段调度中枢规则

第一阶段调度中枢只服务唯一主链，不追求全公司自由智能编排。

第一阶段固定主链：

`business_command -> product -> operations -> risk -> review`

第一阶段调度中枢固定只做下面 6 件事：

1. 判断老板命令是否具备最小可执行条件
2. 判断命令是否属于第一阶段主链范围
3. 按固定主链拆成部门任务
4. 决定任务先进入哪个部门、再落到哪个负责人
5. 判断上游是否达到下游可承接标准
6. 在需要时发起老板拍板或回流

### 5.8.1 输入规则

第一阶段至少要求老板输入说清下面几件事：

1. 目标是什么
2. 最终要交付什么
3. 优先级是什么
4. 时间边界是什么，或者明确当前没有时间边界

缺少关键信息时，不继续硬拆，先回老板补充。

### 5.8.2 拆分规则

第一阶段不做自由动态路由，先做固定主链拆分。

规则是：

1. 内部产品改造类命令，固定先走 `product`
2. `product` 达标后，才允许进入 `operations`
3. `operations` 达标后，才允许进入 `risk`
4. 风控审核结束后，才允许进入当前命令复盘

### 5.8.3 分派规则

第一阶段任务分派固定遵守下面规则：

1. 先入部门公共队列
2. 再落到单一负责人
3. 同一时刻只能有一个当前负责人
4. 可有协作者，但协作者不替代负责人
5. 老板始终保留接管和重新分派权

### 5.8.4 对接规则

第一阶段部门对接不靠感觉，靠门槛。

规则是：

1. `product` 没有完成本段交付物、范围说明、关键验证信息时，`operations` 不准接
2. `operations` 没有完成交付记录、验收材料、异常说明时，`risk` 不准接
3. 下游拒收时，必须明确缺什么，并回流到上游补件

### 5.8.5 拍板规则

第一阶段至少保留下面几类老板拍板节点：

1. 任务拆分后正式下发前
2. 执行中出现重大范围变化时
3. 高风险事项是否继续推进时
4. 风控驳回后是否继续推进时

### 5.8.6 回流规则

第一阶段至少支持三类回流：

1. 回老板补充命令信息
2. 回当前部门补件或重做
3. 回上游部门重新交接

## 6. 第一阶段前端优先顺序

第一阶段前端顺序必须固定，不能跳。

### P1-F0：前端信息架构冻结

目标：

1. 明确一级界面
2. 明确一级界面职责
3. 明确哪些页面冻结

产出：

1. 一级界面清单
2. 界面跳转关系
3. 冻结资产清单

### P1-F1：老板主界面收口

目标：

1. 把老板主界面收成真正第一入口
2. 固定底部聊天输入框
3. 固定主区域只围绕当前命令展开
4. 固定“拆分 -> 预览 -> 确认 -> 下发 -> 进度 / 关键日志”主流程
5. 固定右侧待处理 / 待老板决定操作台

### P1-F2：员工工作台收口

目标：

1. 员工真正能收任务
2. 员工真正能补信息
3. 员工真正能提成果和推进状态

### P1-F3：一期部门页统一模板

目标：

1. 三个部门页结构统一
2. 不再各长各的味道

### P1-F4：研发工作台接执行器

目标：

1. 真正跑 task package -> dry-run -> real-run -> result package

### P1-F5：交付工作台接交接链

目标：

1. product 输出能被 operations 承接

### P1-F6：风控工作台接审批链

目标：

1. operations 输出能被 risk 审批和驳回

## 7. 第一阶段后端最小支撑范围

虽然第一阶段前端先收骨架，但后端必须同步补最小支撑，不然前端就是空壳。

必须同步稳定的后端能力：

1. `business_command` 创建和读取
2. `department_work_item` 拆解和状态推进
3. `first-phase-workspace` 聚合接口
4. `adapter` 注册和查询
5. `adapter_run` 计划、preview、dry-run、real-run、结果回收
6. `dispatch_handoff` 创建、接收、退回
7. `risk_review` 提交、通过、驳回、补件
8. `timeline / audit` 聚合读取
9. `user_account / team_member / member_department_permission` 最小可运行
10. 任务的部门队列、单一负责人、接管 / 转派能力

## 10. 第一阶段模块总顺序

下面这个顺序就是最近真实施工顺序。

| 模块 | 名称 | 一句话目标 |
| --- | --- | --- |
| M0 | 前端信息架构冻结 | 定死一级界面、主入口和冻结区 |
| M1 | 老板主界面骨架 | 收口聊天驱动命令页的最小指挥框架 |
| M2 | 员工工作台骨架 | 收口任务收件箱、归属关系和回写入口 |
| M3 | 一期部门工作台模板 | 统一 product / operations / risk 的页面骨架 |
| M4 | business_command 创建与预览 | 老板可输入、预览、下发 |
| M5 | 核心对象与状态机 | 固定一期对象名和状态流 |
| M6 | 任务拆解与部门路由 | `business_command` 拆到 product / operations / risk |
| M7 | 执行器注册与 Claude 接通 | 先把 `claude-code` 跑通 |
| M8 | product 执行链 | 任务包、dry-run、real-run、结果包 |
| M9 | operations 交接链 | 接 handoff、建交付记录 |
| M10 | risk 审批链 | 审批、驳回、补件、回流 |
| M11 | timeline / audit / review | 把整条链串成可复盘系统 |
| M12 | 可选 Codex 第二执行器 | 只有主链稳后才补 |

## 11. 第一阶段模块详细说明

### 11.1 M0：前端信息架构冻结

目标：

1. 定死一级界面
2. 定死主入口
3. 定死冻结区

输入：

1. 当前前端工程
2. 第一阶段主链定义

输出：

1. 一级界面清单
2. 页面职责表
3. 冻结界面清单

明确不做：

1. 改样式细节
2. 扩业务流程

完成定义：

1. 任何人都能一句话说清第一阶段前端结构

### 11.2 M1：老板主界面骨架

目标：

1. 收成聊天驱动命令页
2. 固定当前命令主视角
3. 固定右侧待老板决定操作台

输入：

1. 当前老板入口页面
2. `firstPhaseWorkspace`

输出：

1. 可输入命令
2. 可看任务拆分与下发预览
3. 可看当前进度和关键日志
4. 可处理待决定事项

明确不做：

1. 完整销售老板台
2. 增长老板台
3. 全局 BI 总览屏

完成定义：

1. 老板只靠主界面就能推进当前命令；需要细节时进入部门页查看

### 11.3 M2：员工工作台骨架

目标：

1. 收成统一执行入口
2. 收成“我负责什么 / 我能看什么 / 我现在该做什么”的个人入口

输出：

1. 待办
2. 补信息
3. 提成果
4. 推状态
5. 清楚显示当前任务是否归我负责

完成定义：

1. 员工不需要跳多个页面就能完成主链动作
2. 同部门多人时，不会出现“大家都能看，但没人真负责”的情况

### 11.4 M3：一期部门工作台模板

目标：

1. 定义统一部门页模板

输出：

1. `Brief`
2. `Pipeline`
3. `Execution`
4. `Artifacts`
5. `Review`

完成定义：

1. product / operations / risk 三页结构一致

### 11.5 M4：business_command 创建与预览

目标：

1. 老板能下命令
2. 老板能预览调度
3. 老板能批准下发

完成定义：

1. 从老板输入到生成预览链路稳定

### 11.6 M5：核心对象与状态机

目标：

1. 固定状态名
2. 固定合法流转
3. 固定前后端语义
4. 固定老板、成员、部门、任务归属之间的最小关系

完成定义：

1. 不同模块不再发明不同状态名
2. 老板权限、部门权限、任务负责人不再混成一层

### 11.7 M6：任务拆解与部门路由

目标：

1. `business_command` 稳定拆成 `product / operations / risk`

完成定义：

1. 一条 `business_command` 能稳定生成一期主链 work items

### 11.8 M7：执行器注册与 Claude 接通

目标：

1. 统一注册执行器
2. 先打通 `claude-code`

完成定义：

1. 平台能发起一次真 `CLI` 调用

### 11.9 M8：product 执行链

目标：

1. 研发工作台接任务
2. 生成任务包
3. preview / dry-run / real-run
4. 结果回收

完成定义：

1. product 页能看到任务包、日志、结果包、产物

### 11.10 M9：operations 交接链

目标：

1. 承接 product 输出
2. 形成交接记录和验收记录

### 15.2 前端验收重点

第一阶段前端验收只看：

1. 主入口是否清晰
2. 主链是否能走通
3. 老板是否能理解当前命令状态
4. 员工是否能完成本职动作
5. 三个部门页是否结构统一
6. 一人公司模式是否可直接工作
7. 同部门多人时是否能清楚看出谁当前负责哪个任务

不看：

1. 页面数量
2. 炫不炫
3. 销售系统做得多不多

## 16. 第一阶段通过标准

第一阶段是否完成，只看下面 13 条是否全部成立：

1. 前端一级界面已经冻结
2. 老板主界面已成为第一入口
3. 员工工作台已成为统一执行入口
4. `product / operations / risk` 三页已统一成部门模板
5. 能创建一个老板目标
6. 能生成标准任务包
7. 能调用至少一个真 `CLI` 执行器
8. 能回收执行结果、日志、产物和失败原因
9. 能形成交付验收记录和风控审批回流
10. 调度中枢能编排完整时间线和复盘，老板主界面能读到结果
11. 老板主账号默认可进入全部门，不需要手动给自己授权
12. 老板能创建员工账号并把员工挂到一期部门
13. 同部门多人时，任务能先入部门队列，再落到单一负责人

缺一条，都不算第一阶段完成。

---

## 17. 最后一句现实话

第一阶段最重要的不是把所有界面都做满，而是把前端主骨架和最小内部闭环收成一套真能工作的系统。只要这一阶段收成了，后面销售、增长、复杂适配器都只是往上扩的问题；如果这一阶段收不成，后面继续堆页面只会更乱。
