## 0. 文档职责

这份文档不是产品介绍，也不是愿景叙事。

它只解决一个内部工程问题：

> 这个项目应该按什么顺序落地，哪些必须先做，哪些坚决不能偷跑，以及每一轮 `vibe coding` 应该怎么收口。

以后它承担 7 个职责：

1. 固定总阶段顺序
2. 固定每阶段目标
3. 固定每阶段进入条件和退出条件
4. 固定前端、对象、流程、执行器的正确施工顺序
5. 固定 `vibe coding` 工作方法
6. 固定禁止事项
7. 固定阶段切换口径

一句话总结：

> 这份文档不是告诉你“梦想有多大”，而是告诉你“先打哪一仗，打完这仗才能打哪一仗”。

## 1. 当前项目最核心的判断

当前项目最大的风险不是“代码少”，而是“顺序乱”。

问题已经很明确：

1. 页面展开得太早，平台内核没先收紧。
2. 销售前端过重，已经反向影响整体节奏。
3. 单轮提示词目标太大，模型只能给局部补丁。
4. 完成定义不先写，导致每轮都像在糊泥。
5. 前端、状态、流程、执行器混写，后期扩展会越来越脏。
6. 阶段边界不硬，导致未来内容不断偷跑到当前。

所以后面必须换成下面这个总路线：

1. 先收前端骨架和信息架构
2. 再收核心对象和状态机
3. 再收主链流程
4. 再收执行器
5. 再收部门细节
6. 最后才扩销售、增长和复杂适配器

## 3. 总施工原则

### 3.1 前端先定骨架，不先堆细节

第一阶段要先把前端方向定住，但“前端先行”必须被正确理解。

正确理解：

1. 先定老板主界面、员工工作台、一期部门工作台这三类入口
2. 先定老板主界面是“聊天驱动的命令页”，不是传统总览大屏
3. 先定一级界面和信息架构
4. 先定部门页统一模板，并明确部门页承接细节下钻

错误理解：

1. 先把所有业务页面做完
2. 先把视觉做满
3. 先把销售和增长整套都铺开

### 3.2 一次只推进一个闭环

一轮只准推进一条清晰路径，例如：

1. `创建 business_command -> 预览 -> 下发`
2. `task package -> dry-run -> real-run -> result package`
3. `product -> operations -> risk`

不允许一轮同时推进：

1. 新部门
2. 新执行器
3. 新审批机制
4. 新对象体系

### 3.3 页面永远服从对象和流程

固定顺序：

`信息架构 -> 对象 -> 状态 -> 服务 -> API -> ViewModel -> 页面 -> 测试 -> 文档`

页面不是第一真相源。

### 3.4 不准跨阶段偷跑

只要当前阶段没过，就不允许为了“更完整”偷偷带入下一阶段内容。

### 3.5 先收主链，再谈丰满

先保证主链成立，再谈：

1. 更复杂的部门行为
2. 更复杂的视觉
3. 更复杂的适配器
4. 更复杂的组织模型

## 4. 前端在总顺序中的地位

这部分必须单独说清，因为现在最容易误判。

### 4.1 前端不是附属品

前端不是最后再补的壳。

没有稳定前端骨架，后面所有流程都没有稳定承载面：

1. 老板没法输入和拍板
2. 员工没法执行和回写
3. 部门没法承接对象和状态

### 4.2 但前端也不是先把所有页面做完

前端应该先做的是：

1. 一级界面定义
2. 老板主界面的操作方式定义
3. 页面职责边界
4. 模块模板统一
5. 与主链一致的最小交互

不应该先做的是：

1. 每个部门的深度业务细节
2. 销售 CRM 式大前台
3. 增长全流水线
4. 大量视觉重绘

### 4.3 第一阶段前端正确顺序

第一阶段前端顺序固定为：

1. 老板主界面信息架构
2. 老板主界面的聊天驱动命令流
3. 右侧待处理 / 待老板决定操作台
4. 员工工作台信息架构
5. 一期部门工作台统一模板
6. 研发工作台接执行器主链
7. 交付工作台接交接主链
8. 风控工作台接审批主链

销售和增长不参与决定第一阶段前端总体结构。

老板主界面在第一阶段必须遵守下面这条结构原则：

1. 底部保留聊天输入框
2. 主区域只围绕当前命令展开
3. 先显示任务拆分和分配确认
4. 再显示当前命令进度和关键日志
5. 右侧固定待处理和待决策操作区

更细的业务对象、证据、交接、审批材料，全部下沉到部门工作台查看。

## 5. 内部分层模型

系统内部统一按 6 层理解，并且这 6 层和开发顺序绑定。

### 5.1 第一层：前端壳层

职责：

1. 一级界面切换
2. 角色模式切换
3. 导航结构
4. 老板主界面的命令输入壳
5. 老板主界面的当前命令展示壳
6. 右侧待处理操作区壳
7. 设置入口

关键要求：

1. 不承载复杂业务规则
2. 不直接写部门状态机
3. 不把全部部门细节堆在老板主界面
4. 老板主界面只承载当前命令，不承载全公司细节浏览

### 5.2 第二层：工作台装配层

职责：

1. 决定哪个工作台挂哪些面板
2. 决定主链默认部门顺序
3. 决定哪些部门当前可见
4. 决定老板主界面如何承接调度中枢输出

### 5.3 第三层：业务对象层

职责：

1. 统一业务语义
2. 定义对象字段和状态
3. 让前后端讲同一套语言

第一阶段必须稳定的对象：

1. `business_command`
2. `department_work_item`
3. `owner_decision`
4. `human_question`
5. `adapter_run`
6. `dispatch_handoff`
7. `department_asset`
8. `risk_review`

### 5.4 第四层：流程与决策层

职责：

1. 任务拆解
2. 部门路由
3. 审批挂点
4. 状态流转
5. 失败回流
6. 复盘触发

这一层里的核心就是调度中枢。

这里必须明确：

1. 调度中枢不是页面
2. 调度中枢不是老板工作台
3. 调度中枢不是普通部门页
4. 调度中枢是系统内部的编排与决策内核

它负责：

1. 把老板指令转成 `business_command`
2. 决定任务怎么拆
3. 决定任务怎么分配和交接
4. 决定各部门在什么阶段该做什么、不该做什么
5. 决定什么节点需要老板拍板

第一阶段不要把调度中枢理解成“开放式智能调度器”，而要理解成“保守、可预测、可回流的规则编排层”。

这一层实现时，先固定下面 6 类规则：

1. 输入规则
   - 判断老板命令是否已经具备最小可执行条件
2. 分类规则
   - 判断命令属于哪类工作
3. 拆分规则
   - 按固定主链生成部门任务
4. 分派规则
   - 先入部门队列，再落到单一负责人
5. 对接规则
   - 上游达到门槛后，下游才允许承接
6. 拍板与回流规则
   - 高风险节点找老板，达不到门槛就回流

第一阶段这里明确不做：

1. 自由动态路由
2. 复杂智能负载均衡派单
3. 全部门自由组合编排
4. 没有门槛的自动强推流转

### 5.5 第五层：执行与适配器层

职责：

1. 执行器注册
2. 任务包生成
3. dry-run / real-run
4. 日志回收
5. 结果归一化

### 5.6 第六层：基础设施层

职责：

1. 身份
2. 权限
3. 存储
4. 配置
5. 日志
6. 审计
7. 用户、团队成员、部门权限和任务归属关系

这一层里，账号和协同机制必须按下面顺序拆开实现：

1. `user_account`
   - 只负责登录认证
   - 不直接代表部门身份
2. `team_member`
   - 代表用户在团队里的内部身份
   - 例如老板、研发负责人、交付专员
3. `member_department_permission`
   - 代表成员能进哪些部门、能做哪些系统动作
4. `task_assignment`
   - 代表当前具体任务归谁执行、谁审核、谁协作

不要把“账号”“部门归属”“任务负责人”写成一层。

## 7. P0：止血和收口

### 7.1 P0 目标

`P0` 不做大功能，只做收口：

1. 冻结第一阶段范围
2. 冻结前端信息架构
3. 冻结核心对象
4. 冻结接口清单
5. 冻结 smoke 清单
6. 标记越界资产

### 7.2 为什么 P0 必须先做

因为如果不先做 `P0`：

1. 你会继续一边写一边改目标
2. 销售会继续吞主线
3. 前端会继续同时服务三套方向
4. 后端接口和前端交互会持续漂移

### 7.3 P0 明确不做什么

1. 不做大重构
2. 不做页面重绘
3. 不扩新部门
4. 不扩新执行器

### 7.4 P0 必须产出

1. 一期前端信息架构图
2. 一期一级界面清单
3. 一期核心对象清单
4. 一期接口清单
5. 一期 smoke 清单
6. 一期冻结资产清单

### 7.5 P0 进入条件

无，当前阶段默认从 `P0` 开始。

### 7.6 P0 退出条件

必须全部成立：

1. 能一句话说清第一阶段只做什么
2. 能画出第一阶段前端一级界面
3. 能列出一期主链对象
4. 能列出一期必需接口
5. 能列出一期 smoke
6. 能指出哪些代码是冻结资产

## 8. P1：最小内部闭环

### 8.1 P1 目标

跑通这条链：

`老板目标 -> 调度中枢编排 -> 研发任务包 -> CLI 执行 -> 交付交接 -> 风控审批 -> 调度复盘`

### 8.2 为什么 P1 先做这个

因为这条链同时验证 4 件事：

1. 调度能力
2. 执行器能力
3. 审批能力
4. 结果回收能力

### 8.3 P1 明确不做什么

1. 真销售闭环
2. 真增长闭环
3. `MCP`
4. Browser runner
5. GUI 自动化
6. 多租户
7. 企业化部署

### 8.4 P1 必须产出

1. 老板主界面主链可用
2. 员工工作台主链可用
3. `product / operations / risk` 统一模板工作台可用
4. 至少一个真 `CLI` 执行器跑通
5. 时间线和复盘可读
6. 老板主账号、员工子账号、部门权限和单任务负责人机制可用

1. `P0` 已通过
2. 一期前端骨架已冻结
3. 一期对象、接口、smoke 已冻结

### 8.6 P1 退出条件

必须全部成立：

1. 能创建 `business_command`
2. 能生成标准 `department_work_item`
3. 能至少跑一个真 `CLI` 执行器
4. 能回收日志、产物、失败原因
5. 能形成交付记录
6. 能形成风控审批和驳回回流
7. 能形成完整时间线和复盘
8. 老板不需要给自己手动分配权限也能进入全部门
9. 老板能创建员工账号并把员工挂到一个或多个部门
10. 同部门多人时，任务能先入部门队列再落到单一负责人

## 13. 账号与协同实现原则

这部分必须单独定死，不然后面最容易越写越乱。

先把三个概念彻底分开：

1. `老板主界面`
   - 前端入口
   - 负责老板输入、确认、查看当前命令状态
2. `调度中枢`
   - 业务编排内核
   - 负责拆解、分派、协同、阶段控制、审批挂点
3. `business_command`
   - 调度中枢管理的核心业务对象

### 13.1 老板不是被分配权限的人

老板主账号默认就是全团队最高权限。

固定规则：

1. 老板默认可看全部门
2. 老板默认可进全部门
3. 老板默认可接管全部门任务
4. 老板默认可重新分派任务
5. 老板默认可审批和回看全部日志

不要做成“老板先给自己勾选所有部门权限”的绕路设计。

### 13.2 三种使用模式必须同时兼容

实现时必须同时兼容：

1. `solo_mode`
   - 老板一个人干全部门
2. `delegated_mode`
   - 老板把部门委派给员工
3. `hybrid_mode`
   - 老板自己兼一部分，另一部分给员工

其中 `hybrid_mode` 应该按默认现实场景来设计。

### 13.3 同部门多人协同的最小规则

不要一开始就做成多人无限制共改。

先固定下面这套最小规则：

1. 一个部门只有一个主负责人
2. 一个部门可以有多个执行成员
3. 一个部门可以有审核成员
4. 一个任务进入执行态时只能有一个当前负责人
5. 任务可以有人协作，但协作者不替代负责人
6. 审核人负责通过或退回，不直接代替负责人写主内容

### 13.4 第一阶段只做最小协同，不做完整企业权限中心

第一阶段只把下面这些做成：

1. 老板注册 / 登录
2. 老板创建员工账号
3. 员工归属一个或多个部门
4. 部门队列 -> 单一负责人 -> 审核 / 回流
5. 老板接管 / 转派 / 拍板

第一阶段不做：

1. SSO
2. MFA
3. 完整邀请激活流程
4. 复杂角色模板中心
5. 复杂跨组织治理

## 14. 总阶段禁止事项

只要还没进对应阶段，下列事情一律禁止：

1. 第一阶段继续扩销售前端
2. 第一阶段接增长全链路
3. 第一阶段接 `Cursor` 主链、`MCP`、Browser、GUI
4. 第一阶段大面积重做视觉
5. 为了“看起来高级”而重构全部壳层
6. 阶段未过就吹完整公司系统已成型

## 15. 每轮 vibe coding 的固定顺序

每轮都必须按这个顺序：

1. 先确认当前阶段
2. 先确认当前模块
3. 先写输入输出
4. 先写本轮不做什么
5. 先列修改文件
6. 先列验证方式
7. 再让模型写

如果做不到，就说明这一轮范围没收住。
