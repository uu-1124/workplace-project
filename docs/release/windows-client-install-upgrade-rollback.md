# Windows 客户端安装、升级与回滚手册

## 1. 适用范围

本文面向实施和一线支持人员，适用于 Windows x64 桌面端 `SoloOps Console` 当前第一阶段交付链路。

当前已自动演练的版本路径：

```text
2.0.0 -> 2.0.1 -> 故障检测 -> 2.0.0 回滚
```

本手册不代表产品已经达到正式 GA。当前安装器仍未完成正式代码签名，统一干净 Windows runner 的升级门禁也必须获得远端绿灯后，才能把此流程作为正式客户发布承诺。

## 2. 交付前必须具备的材料

实施人员开始操作前必须同时拿到：

1. 当前根项目的 `release-manifest.json`，并确认 `release_id`。
2. 当前版本安装器：`SoloOps Console Setup 2.0.1.exe`。
3. 上一个可回滚版本安装器：`SoloOps Console Setup 2.0.0.exe`。
4. 本次发布说明及已知限制。
5. 客户后端地址、团队信息和负责本次变更的支持联系人。

不得只保存当前版本安装器而丢弃回滚版本。不得使用文件名不明、来源不明或版本号与发布清单不一致的安装包。

## 3. 客户状态边界

桌面端的后端地址、团队、账户模式、工作台和主题等客户端状态由 Electron/Chromium 保存在用户数据目录中，默认位于 Windows 当前用户的 AppData 范围，而不是应用安装目录。

因此：

- 覆盖升级和回滚必须安装到原安装目录。
- 升级失败时不得先手工删除 AppData 或未知目录。
- 卸载应用不等于获得了删除客户状态的授权。
- 如客户要求彻底清除本地状态，必须作为单独的数据清理操作确认，不能混入普通卸载步骤。

## 4. 首次安装

1. 确认旧版 `SoloOps Console` 未在运行。
2. 运行当前版本安装器。
3. 使用默认安装目录；如客户必须自定义目录，应记录该目录，后续升级和回滚必须继续使用同一路径。
4. 启动客户端。
5. 在登录首屏或设置页确认：
   - 平台服务可达；
   - 客户端/API 协议兼容；
   - 数据库状态正常；
   - 团队访问正常；
   - 模型状态符合本次交付预期。
6. 若检查失败，不要反复重装。先导出脱敏诊断，再按后端、协议、数据库、团队或模型状态分类处理。

## 5. 覆盖升级

### 5.1 升级前

1. 记录当前客户端版本、API 版本和 `release_id`。
2. 从设置页导出一次脱敏诊断。
3. 确认上一个可回滚版本安装器在本机或受控交付目录中可用。
4. 退出客户端，并确认没有正在提交的重要操作。
5. 不卸载旧版本，不删除用户数据目录。

### 5.2 执行升级

1. 运行 `SoloOps Console Setup 2.0.1.exe`。
2. 选择原安装目录完成覆盖安装。
3. 启动客户端。
4. 重新执行首启检查，并核对升级前已有的后端地址、团队、账户模式、工作台和主题没有丢失或异常重置。
5. 执行一条不会产生不可逆业务影响的第一阶段主链检查，确认客户端可以正常读取服务端状态。

## 6. 升级失败判断

出现以下任一情况，应停止继续操作并进入恢复流程：

- 客户端无法启动或启动后白屏；
- 安装后实际版本与目标版本不一致；
- 首启检查显示协议不兼容；
- 原有客户端状态丢失或被异常重置；
- 基础主链检查无法完成，且已确认不是后端现场故障。

在进入回滚前，记录错误时间、目标版本、原版本、安装目录、可见错误摘要和脱敏诊断。不要通过删除目录“试试看”。

## 7. 失败恢复与回滚

1. 退出当前客户端。
2. 保留故障现场和用户数据，不先卸载当前版本。
3. 运行固定的上一个版本安装器 `SoloOps Console Setup 2.0.0.exe`。
4. 选择与升级前相同的安装目录，执行原位覆盖回滚。
5. 启动客户端，确认实际版本恢复为 `2.0.0`。
6. 核对后端地址、团队、账户模式、工作台、主题等升级前状态仍可读取。
7. 重新执行平台、协议、数据库和团队访问检查。
8. 回滚成功后停止再次升级，提交故障诊断，由发布负责人决定是否撤回候选发布。

如果旧版本安装器无法覆盖恢复，或回滚后客户状态仍不一致，应立即升级给研发支持；此时仍不得删除 AppData。当前流程不承诺跨数据库破坏性迁移的自动恢复，涉及服务端数据库迁移时必须同时执行对应的服务端回滚方案。

## 8. 卸载

1. 退出客户端。
2. 运行 `Uninstall SoloOps Console.exe` 或 Windows 应用卸载入口。
3. 确认安装目录中的程序文件已清理。
4. 默认保留用户数据，便于重装和故障恢复。
5. 只有客户明确授权彻底清除本地状态时，才由支持人员执行单独的数据清理流程。

## 9. 自动化验收门禁

桌面仓库本地门禁：

```powershell
npm run build
npm run test:installer:lifecycle
npm run test:installer:upgrade
```

升级门禁会执行：

```text
安装固定旧版
-> 写入隔离客户状态
-> 原位升级当前版
-> 启动验证
-> 状态保留验证
-> 注入 app.asar 缺失故障并确认门禁真实失败
-> 原位安装固定旧版回滚
-> 启动验证
-> 状态再次保留验证
-> 静默卸载并检查程序文件清理
```

Desktop 仓库和根项目 Windows release workflow 都必须从固定提交 `7dda20c047627ac714898f08b18ca9787f000cf5` 重新构建 `2.0.0` 安装器，不允许依赖 runner 或开发机残留的旧安装包。

## 10. 当前未关闭限制

- 正式代码签名尚未完成。
- 新增升级门禁尚需取得干净 Windows runner 远端绿灯。
- 当前只演练 `2.0.0 -> 2.0.1` 的 Windows 单机 NSIS 原位升级和回滚。
- 未引入自动更新服务、增量更新或后台静默更新。
- 服务端数据库迁移回滚不属于本客户端手册的覆盖范围。
- 在上述限制关闭前，不得对客户宣称“无感自动升级”或“任意版本均可自动回滚”。
