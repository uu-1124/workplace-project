# 企业级部门热插拔实施指南（最终蓝图）

## 1. 文档定位

这份文档只负责回答三个问题：

1. 这个产品最终要成为什么。
2. 平台、部门、人、外部软件之间分别负责什么。
3. 整个系统长期应该按什么形态演进。

这不是当前迭代的施工说明书，不负责：

1. 本轮具体改哪些文件。
2. 本周做哪个模块。
3. 某个接口字段怎么定义。
4. 第一阶段的唯一验收标准。

这份文档的使用方式很明确：

- 对外：统一叙事和产品定位
- 对内：统一长期边界和最终形态
- 对未来：为后续阶段预留扩展位

当前文档治理只保留两份活跃文档：

1. 本文：定义长期产品方向、最终边界和后续扩展位。
2. `上线就绪整改计划.md`：定义当前唯一允许执行的整改项、顺序、验收标准和冻结范围。

`docs/archive/` 下的第一阶段计划、工程规格、扫描结论和台账只保留为历史证据，不能直接派发当前开发任务。

---

## 2. 产品一句话定义

这不是一个“平台自己包打天下”的超级 AI 员工系统。

这是一个：

> 以经营流程为骨架、以部门工作台为界面、以热插拔外部软件为执行层、以审批和审计为约束的企业级 AI 部门操作系统。

它卖的不是“模型更强”，而是：

1. 组织流程能力
2. 任务编排能力
3. 上下文打包能力
4. 外部软件调度能力
5. 结果回收能力
6. 审批和复盘能力

---

## 3. 平台和外部软件的边界

### 3.1 平台负责什么

平台负责：

1. 接收老板目标和经营指令。
2. 把目标拆成跨部门任务。
3. 为每个部门生成上下文、边界、材料和验收口径。
4. 统一管理部门工作台、状态流、审批流和交接流。
5. 调用外部软件执行具体工作。
6. 回收日志、产物、风险、状态和复盘结论。
7. 形成完整经营时间线和审计记录。

### 3.2 平台不负责什么

平台不负责：

1. 重做 `Claude Code`、`Codex`、`Cursor`
2. 重做 CRM、广告平台、项目管理软件、知识库、飞书
3. 第一阶段就替代企业全部专业软件
4. 试图让单个智能体独立完成整个公司经营

### 3.3 外部软件负责什么

外部软件负责：

1. 写代码
2. 生成内容
3. 执行触达
4. 管理项目
5. 提供结构化业务能力

平台调用它们，但不假装自己就是它们。

---

## 4. 核心原则

1. 平台管流程，不抢专业工具的活。
2. 能接现成软件，就不重复造轮子。
3. 所有部门能力必须可替换、可禁用、可升级。
4. 所有关键执行结果必须可追踪、可审计、可复盘。
5. 关键节点必须允许人工拍板。
6. 任何部门都不能变成黑盒页面。
7. 产品范围必须小于想象力，否则工程一定失焦。
8. 前端要承载经营过程，不是做漂亮壳子。

---

## 5. 最终经营闭环

成熟形态下，平台支持的完整经营闭环应为：

1. 老板输入经营目标
2. 调度中枢拆解任务和优先级
3. 市场增长部产出流量和线索
4. 销售成交部把线索推进成订单和合同
5. 财务法务风控部控制预算、价格、合同、回款和合规
6. 研发工程部补产品、补工具、补能力
7. 交付运营部完成交付、培训、验收、续费
8. 所有结果回流调度中枢形成复盘
9. 下一轮目标再次下发

注意：

- 这是最终目标形态
- 不是第一阶段施工范围
- 第一阶段只证明平台最小内核成立

## 6. 前端最终产品形态

从产品视角看，前端最终应收成 4 类界面，而不是一堆散装页面。

这里有一个非常关键的设计原则：

> 老板主界面负责轻指挥和当前命令推进，不负责承载所有细节；  
> 具体业务细节、对象细节、证据细节，全部下沉到部门工作台查看。

### 6.1 老板主界面

老板主界面是第一主入口，但它不应该做成重后台，也不应该做成大而全总览屏。

它的正确形态是：

> 一个聊天驱动的命令页。

界面组织原则如下：

1. 底部保留一个像常见 AI 工具那样的对话输入框。
2. 对话框负责输入目标、补充边界、调整方向、追问和重新下令。
3. 主区域只围绕“当前命令”展开，不同时承载一堆无关全局信息。
4. 当命令发出后，主区域依次展示：
   - 系统正在拆分任务
   - 拆分后的任务分配结果
   - 老板确认后开始分配任务
   - 执行中的当前进度和关键日志
5. 右侧固定保留“待处理 / 待老板决定”的操作台。

老板主界面长期只负责下面这些事情：

1. 下命令
2. 看当前命令进行到哪
3. 看最近关键日志
4. 看是否出现待拍板、待补充、待驳回事项
5. 在必要时进入部门页查看细节

它不是某个部门页，也不是传统 BI 大屏，更不是把所有部门细节都堆在一起的总览后台。

### 6.2 员工工作台

员工工作台负责：

1. 接收个人待办
2. 回复补充问题
3. 查看依赖和阻塞
4. 提交产物
5. 推进状态

它是“按人组织”的执行台，不承担老板级全局指挥。

### 6.3 部门工作台

部门工作台负责承载部门业务对象、部门主链和细节下钻。

老板主界面不应该塞进去的内容，都应该在这里看：

1. 具体任务包
2. 具体执行器运行细节
3. 具体交接内容
4. 具体审批材料
5. 具体结果包、证据和异常

不同部门有不同内容，但结构必须统一。

每个部门工作台最终都要围绕下面 5 个标准模块展开：

1. `Brief`
2. `Pipeline`
3. `Execution`
4. `Artifacts`
5. `Review`

### 6.4 系统设置层

系统设置层负责：

1. 模型配置
2. 后端环境
3. 团队参数
4. 运行状态

它不应和业务界面混在一起。

## 7. 部门蓝图

### 7.1 调度中枢

职责：

1. 接收老板目标
2. 生成 `business_command`
3. 拆解部门任务
4. 路由部门
5. 汇总风险
6. 发起审批
7. 汇总时间线和复盘

它是平台内核，不是普通部门页，也不是前端可视化首页。

这里必须明确区分三件事：

1. `老板主界面`
   - 是老板下命令、看当前命令推进、做拍板的前端入口
2. `调度中枢`
   - 是系统内部的编排与决策内核
   - 负责拆解、分派、协同、阶段控制、审批挂点和回流规则
3. `business_command`
   - 是调度中枢管理的核心业务对象
   - 不是页面名，也不是普通部门页名

调度中枢长期不追求“自由智能乱编排”，而追求“规则清楚、阶段清楚、责任清楚”。

### 7.1.1 调度中枢规则层

调度中枢长期至少要稳定回答下面 6 个问题：

1. 这条老板命令是否已经具备最小可执行条件
2. 这条命令应该进入哪条部门主链
3. 这条命令应该拆成哪些部门任务
4. 每个任务应该先进入哪个部门、交给谁负责
5. 上游做到什么程度，下游才允许承接
6. 哪些节点必须老板拍板，哪些节点必须回流

### 7.1.2 输入规则

调度中枢先判断老板输入是否具备最小执行条件。

长期最小输入原则：

1. 必须说清目标
2. 必须说清最终要交付什么
3. 必须说清优先级
4. 必须说清时间边界，或者明确当前没有时间边界
5. 如果关键信息缺失，不允许假装理解，必须回老板补充

### 7.1.3 拆分规则

调度中枢先按固定主链拆分，不按无限自由路由拆分。

第一阶段长期固定主链：

`business_command -> product -> operations -> risk -> review`

长期拆分原则：

1. 先识别命令属于哪类工作
2. 再决定需要经过哪些部门
3. 每个部门必须拿到自己的目标、输入、输出和对接对象
4. 不允许只写“把这个给某部门”，不写承接边界

### 7.1.4 分派规则

调度中枢决定任务归属时，长期遵守下面规则：

1. 先入部门，再入个人
2. 任务先进入部门公共队列
3. 进入执行态前，必须明确一个当前负责人
4. 同一时刻只能有一个主负责人
5. 老板始终保留接管和重新分派权

### 7.1.5 对接规则

调度中枢必须对“部门之间什么时候允许交接”设硬门槛。

长期原则：

1. 上游没有达标，下游不准硬接
2. 对接标准必须围绕交付物、证据、风险说明、验收材料来定
3. 下游拒收后，必须明确退回到上游哪一段补件
4. 不允许靠口头理解完成流转

### 7.1.6 拍板规则

调度中枢不替老板做最终高风险判断。

长期至少保留下面几类老板拍板节点：

1. 任务拆分完成后是否允许正式下发
2. 执行中是否接受范围变化
3. 高风险事项是否允许继续推进
4. 风控驳回后是否继续推进或重做

### 7.1.7 回流规则

调度中枢必须把回流当成主链的一部分，不是异常补丁。

长期至少支持三类回流：

1. 回老板补信息
2. 回当前部门补件或重做
3. 回上游部门重新交接

### 7.2 研发工程部

职责：

1. 把需求收成边界清晰的任务包
2. 生成架构和实现方案
3. 调用代码类执行器
4. 回收代码、测试、构建和风险说明
5. 输出研发产物供交付承接

这是最适合验证热插拔执行器的第一批部门。

### 7.3 交付运营部

职责：

1. 接收研发或销售上游输出
2. 组织交接、排期、阻塞和验收
3. 形成项目推进记录
4. 形成交付物、培训物和复盘物

它负责把“结果”变成“可交付”。

### 7.4 财务法务风控部

职责：

1. 风险卡口
2. 合同与预算边界
3. 风控审批
4. 驳回重提
5. 合规意见留痕

它不是说明页，而是流程中的强制卡点。

### 7.5 销售成交部

职责：

1. 线索推进
2. 跟进节奏
3. 报价审批协同
4. 签约和回款衔接
5. 成交复盘

商业价值高，但复杂度也高，不适合当前第一阶段主链。

### 7.6 市场增长部

职责：

1. 内容生产
2. 渠道触达
3. 投放和归因
4. 线索沉淀
5. 数据回流

工具最杂，周期最长，必须放在更后阶段。

## 8. 核心业务对象

长期稳定的核心对象必须统一，不允许每个阶段重新发明。

### 8.1 公司级对象

1. `business_command`
2. `owner_decision`
3. `timeline_event`
4. `department_asset`

### 8.2 部门级对象

1. `department_work_item`
2. `dispatch_handoff`
3. `risk_review`
4. `human_question`

### 8.3 执行级对象

1. `adapter_run`
2. `task_package`
3. `result_package`
4. `execution_log`

这些对象是平台长期可演化的骨架，不应随页面波动。

## 9. 账号与协同模型

平台里的“谁登录”“谁属于哪个公司”“谁能进哪个部门”“谁当前对哪个任务负责”必须拆开定义，不能混成一层。

### 9.1 平台术语边界

下面几个词必须长期固定，不允许混用：

1. `老板主界面`
   - 前端入口
   - 负责老板输入命令、确认分派、查看当前命令状态
2. `调度中枢`
   - 平台编排内核
   - 负责拆任务、定路由、控协同、控阶段、控审批
3. `business_command`
   - 老板指令对应的核心业务对象
4. `部门工作台`
   - 承接部门细节执行和结果沉淀

### 9.2 四层账号模型

长期统一按下面四层建模：

1. `user_account`
   - 只负责登录认证
   - 只关心邮箱、密码、会话、安全设置
   - 不直接代表公司身份和部门身份
2. `team_member`
   - 代表用户在某个团队里的内部身份
   - 例如老板、研发负责人、交付专员、风控审核员
   - 一个 `user_account` 可以在不同团队里映射成不同 `team_member`
3. `member_department_permission`
   - 负责定义某个成员能进入哪些部门、以什么权限进入
   - 权限是系统动作，不是岗位描述
4. `task_assignment`
   - 负责定义某个具体任务当前归谁执行、谁协作、谁审核、谁关注
   - 任务归属不能用“部门权限”代替

### 9.3 老板主账号规则

老板主账号不是“被额外授予所有部门权限”，而是天然拥有全团队总权限。

老板默认具备：

1. 全部门可见
2. 全部门可进入
3. 全部门可接管
4. 全部门可转派
5. 全部门可审批
6. 全部门可查看日志、产物和交接记录

这条规则长期固定，不建议做成“老板给自己勾选全部权限”的复杂设计。

### 9.4 三种团队工作模式

平台长期同时支持下面三种模式：

1. `solo_mode`
   - 老板不创建子账号
   - 老板兼任全部门负责人
   - 所有任务都允许直接落到老板自己
2. `delegated_mode`
   - 老板创建员工子账号
   - 把一个或多个部门分配给员工
   - 老板保留总接管权和审批权
3. `hybrid_mode`
   - 老板自己兼任一部分部门
   - 另一部分部门委派给员工
   - 这是最现实的长期默认模式

### 9.5 部门内角色模型

同一个部门里可以有多人，但角色不能混乱。

长期建议只保留四类角色：

1. `department_owner`
   - 每个部门默认只认一个主负责人
   - 负责看部门总队列、分派任务、对外代表本部门交接
2. `department_executor`
   - 可有多个
   - 负责接任务、补材料、产出结果、推进状态
3. `department_reviewer`
   - 可有多个
   - 负责审核、退回、批准进入下游
4. `department_observer`
   - 只查看，不直接改业务对象

部门多人不等于所有人都能随便改所有东西。

### 9.6 任务协同规则

多人同部门协同必须遵守下面这套固定规则：

1. 任务先进入部门公共队列
2. 进入执行态前，必须明确一个当前负责人
3. 一个任务同一时刻只能有一个主负责人
4. 可以有多个协作者，但协作者不等于负责人
5. 可以有审核人，但审核人不等于负责人
6. 老板始终保留接管和重新分配权

长期任务层推荐至少包含：

1. `assigned_member_id`
2. `collaborator_member_ids`
3. `reviewer_member_id`
4. `watcher_member_ids`

### 9.7 多人协同边界

平台不应该默认允许“多人同时任意改同一个对象”。

长期边界建议如下：

1. 负责人负责主内容编辑
2. 协作者负责补充材料、评论、局部输入
3. 审核人负责通过、退回、给审核意见
4. 观察成员负责查看和知会
5. 真正跨人协作优先通过任务拆分和对象分段，而不是多人无边界共改

## 10. 热插拔能力模型

平台里的能力不应暴露成“一堆按钮”，而应暴露成标准适配器。

```json
{
  "adapter_id": "claude-code",
  "adapter_type": "cli",
  "department_scope": ["product"],
  "capabilities": ["code_generation", "test_execution"],
  "input_contract": "task_package_v1",
  "output_contract": "result_package_v1",
  "status": "enabled"
}
```

所有适配器都必须有：

1. 唯一标识
2. 类型定义
3. 能力声明
4. 输入契约
5. 输出契约
6. 日志回收
7. 启停和替换机制

---

## 11. 外部软件接入模式

长期看，平台支持四类接入模式。

### 11.1 CLI

适合：

1. 代码执行器
2. 本地工具
3. 第一阶段主链

### 11.2 API

适合：

1. CRM
2. 广告平台
3. 表单平台
4. 内容服务

### 11.3 Browser

适合：

1. 没有稳定 API 的网页系统
2. 操作链清晰但系统封闭的场景

### 11.4 GUI / Desktop Automation

适合：

1. 桌面软件
2. 老旧系统
3. 极少数必须的人机协作场景

但这类成本最高，不能当第一阶段主轴。

## 12. 技术架构分层

### 12.1 桌面工作台层

建议技术：

- Electron
- React
- TypeScript
- Zustand 或 Redux Toolkit
- TanStack Query

职责：

1. 呈现老板、员工、部门三类工作台
2. 承载输入、查看、审批、提交等交互
3. 不承载复杂业务规则

### 12.2 工作台装配层

职责：

1. 决定不同角色看到什么
2. 决定部门页挂什么模块
3. 决定主链默认顺序和可见性
4. 决定老板模式、员工模式和部门模式如何切换

### 12.3 业务编排层

建议技术：

- FastAPI
- Python
- Pydantic
- SQLAlchemy

职责：

1. 管理业务对象
2. 管理状态机
3. 管理任务拆解和路由
4. 管理审批和复盘
5. 管理部门队列、任务归属和协同边界

### 12.4 执行适配器层

职责：

1. 把平台任务转成软件输入
2. 屏蔽不同软件差异
3. 回收并归一化结果

### 12.5 数据与审计层

建议技术：

- PostgreSQL
- Redis（可选）
- 对象存储

职责：

1. 存业务对象
2. 存日志
3. 存审批记录
4. 存产物元数据
5. 存适配器注册信息
6. 存账号、成员、权限和任务归属关系

## 14. 阶段路线图

### 14.1 P1：最小内部闭环

目标：

1. 证明平台有调度能力
2. 证明平台有热插拔能力
3. 证明平台有审批能力
4. 证明平台有结果回收能力

主链固定为：

`调度中枢 -> 研发工程部 -> 交付运营部 -> 财务法务风控部 -> 调度中枢复盘`

### 14.2 P2：稳定性加固

目标：

1. 状态持久化
2. 失败恢复
3. 权限边界
4. 审计链完整
5. 账号与成员协同边界收口

### 14.3 P3：销售重启

目标：

1. 在稳定底盘上接回简化销售
2. 接回线索、跟进、报价审批和签约衔接

### 14.4 P4：增长和复杂适配器

目标：

1. 增长部门接入
2. API / MCP / Browser 接入
3. 更复杂的软件选择和路由策略

### 14.5 P5：企业化硬壳

目标：

1. 多租户
2. 部署
3. 安装、更新、签名
4. 监控和告警
5. 合规和审计输出
6. 更完整的邀请、找回密码、MFA、组织治理等企业账号能力

## 15. 当前最现实路线

当前主链已经具备受控试点能力，因此最现实路线不再是继续扩第一阶段功能，而是：

1. 先把当前版本收成可复现、可发布的基线。
2. 再解决客户安装、升级、回滚和基础诊断。
3. 再验证 PostgreSQL 与真实模型等生产路径。
4. 再补任务运行底座、稳定运行和客户运营闭环。
5. 只有上述基础成立后，才恢复销售、增长和复杂外部系统的扩展。

原因：

1. 当前核心流程已能运行，真正缺口已转为交付和生产化。
2. 未建立发布基线前，继续扩功能无法稳定复制给客户。
3. 销售最容易把范围拖散。
4. 增长最容易把整改阶段重新做成大量漂亮废件。
