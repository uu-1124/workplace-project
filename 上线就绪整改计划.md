# 上线就绪整改计划

日期：2026-07-12  
状态：活跃 / 当前唯一执行计划

## 1. 文档职责与优先级

本文件是当前唯一可直接派发开发、测试和发布工作的文档。

- `企业级部门热插拔实施指南(最终蓝图).md`：只定义长期方向和最终边界。
- 本文件：定义当前整改范围、执行顺序、验收标准和冻结范围。
- `docs/archive/`：只保留历史决策、第一阶段工程契约和完成证据，不直接产生新任务。

当长期蓝图、历史文档与本文件发生冲突时，以本文件为准。代码改动涉及主链对象、状态机或接口时，必须先核对归档工程规格与现有测试，再更新本文件中的契约摘要。

## 2. 当前判断

当前产品已跑通受控试点所需的核心工作流：

`老板指令 -> product -> operations -> risk -> 审批 / 返工 / 补件 / 交接`

当前目标不是继续增加部门、页面或适配器，而是把已存在的主链变成可复制交付的客户产品。

当前不具备正式企业级 GA 的主要原因：

1. 发布版本和工作树基线不可稳定复现。
2. CI、release gate 与远端环境存在漂移，且现有门禁存在失败。
3. Windows 正式安装器构建失败，升级和回滚机制不完整。
4. 生产 PostgreSQL 路径和真实模型调用未成为持续发布门禁。
5. 后端仍是单活动 API 进程，长任务和 API 请求尚未形成可靠运行底座。
6. 企业客户日常运营所需的账号生命周期、通知、诊断和管理员自助能力不完整。

## 3. 当前范围与冻结规则

### 3.1 允许推进

只允许推进以下四类整改：

1. 发布基线、版本、CI/CD 与可重复构建。
2. 客户端安装、升级、回滚、首启检查和诊断。
3. PostgreSQL、真实模型和隔离环境中的生产路径验证。
4. 任务运行、恢复、监控、账号运营等客户稳定使用能力。

### 3.2 明确冻结

以下内容不得作为当前开发任务，除非本文件更新并完成范围评审：

- 销售工作台及其线索、商机、报价、活动链路扩展。
- 增长工作台、投放、营销素材和渠道自动化扩展。
- 新部门、新 Agent 角色或新业务主链。
- MCP、Browser runner、GUI 自动化和复杂新适配器。
- 为视觉效果进行的前端重绘或大规模组件重构。
- 为“代码更漂亮”进行的无业务价值重构。
- 新计费、支付和订阅功能。

### 3.3 主链工程契约摘要

当前默认主链只能是：

`product -> operations -> risk`

必须继续保持的行为：

- 老板创建、预览、批准并派发业务指令。
- 部门工作项、成果、审批、退回、补件和交接均可追踪。
- `operations` 的已批准成果才可释放 `risk` 审查。
- 老板驳回、补件、返工和失败恢复不能打散原有任务关系。
- CLI/代码执行必须保留任务包、结果包、运行记录和失败原因。
- `sales / growth` 仅可作为冻结兼容入口，不参与默认进度、默认验收或当前整改范围。

### 3.4 发布身份契约

每次可交付发布必须以根目录 `release-manifest.json` 的 `release_id` 为唯一客户、支持和回滚身份。该清单必须同时绑定：根项目源码基线 commit、两个子模块 commit、API 运行版本、数据库 migration head、桌面端 package 版本和默认 release gate。验证器会输出清单实际所在的根项目 commit；不在清单中自引用该 commit。

发布前必须执行：

```powershell
node scripts/verify-release-manifest.mjs
```

不得仅凭 API、Electron 或数据库中的任一单独版本号判断客户运行的发布版本。

## 4. 执行规则

每一个整改项开始前必须回答：

1. 它是否直接解决本文件第 2 节中的一个正式上线缺口？
2. 它是否会改变主链对象、状态机、公共接口或数据库？
3. 是否已有最小验证命令；若没有，能否在本项中补齐？
4. 是否会误把冻结模块重新带入默认入口或验收？

若第 1 题为否，或第 4 题为是，不应开始编码。若第 2 题为是，必须先说明影响范围并保持最小可回滚修改。

## 5. R0：发布基线可复现

### 目标

让任意开发者从一个确定的提交，在干净机器上构建、测试并得到一致的发布物。

### 工作项

1. 清点并处理 `workplace-api`、`workplace-desktop-electron` 的未提交和未跟踪文件。
2. 确保所有实际使用的迁移、脚本、服务和配置模板进入版本管理。
3. 统一 API、桌面端、数据库迁移和发布标签的版本映射。
4. 修复当前失败的后端 smoke 与桌面端 release gate。
5. 移除 release gate 对固定公网、状态不可控环境的硬依赖；改为可重复的隔离验证环境。
6. 在干净 checkout 中验证后端、桌面端、打包后应用和第一阶段 UI 链路。

### 完成标准

- 两个子模块工作树可明确对应发布 commit。
- 干净 checkout 可完成依赖安装、数据库初始化、构建和测试。
- CI 全绿且不依赖人为维护的远端脏环境。
- 发布物可关联到唯一的客户端版本、API 版本、迁移版本和 commit。

## 6. R1：客户可安装、可升级

### 目标

让非研发实施人员可以安装、配置、升级、卸载并收集基础诊断信息。

### 工作项

1. 修复 Windows 安装器构建及其依赖准备流程。
2. 建立标准安装、卸载、升级和失败恢复流程。
3. 定义客户端与服务端版本兼容规则。
4. 在首启流程提供后端连接、团队访问和模型状态检查。
5. 提供面向支持人员的最小诊断信息导出：版本、运行环境、连接状态、任务关联标识和错误摘要。
6. 建立客户端回滚方案，并实际演练至少一次升级失败恢复。

### 完成标准

- Windows 正式安装器可在干净机器构建并安装。
- 客户端可明确提示不兼容版本和连接异常。
- 更新失败不会要求客户手工删除未知目录才能恢复。
- 实施人员仅依据正式文档即可完成首次交付。

## 7. R2：生产路径真实验证

### 目标

让发布门禁验证真实客户路径，而不是仅验证 SQLite、fallback 模板或开发机状态。

### 工作项

1. 为 CI 增加 PostgreSQL 服务，并从零执行迁移和主链回归。
2. 建立 SQLite 与 PostgreSQL 的明确测试边界；生产契约必须在 PostgreSQL 验证。
3. 建立最小真实模型 E2E：连接、超时、失败、限流、格式异常和恢复。
4. 在 UI 和审计中明确区分真实模型结果与 fallback 结果。
5. 构建隔离临时环境，运行桌面端到后端的主链 E2E 和发布后 smoke。
6. 验证一次旧版本升级到新版本的数据库迁移和客户端兼容路径。

### 完成标准

- PostgreSQL 主链测试进入持续 CI。
- release gate 包含真实模型最小 E2E，或在模型依赖不可用时明确失败而非静默 fallback 成功。
- 临时隔离环境中的客户端、服务端和迁移链路可重复通过。
- 生产数据库和模型问题能够在发布前暴露。

## 8. R3：稳定运行与客户运营闭环

### 目标

将当前单实例受控试点系统推进为可稳定运行、可诊断、可恢复的客户产品底座。

### 工作项

1. 将长任务执行从 API 请求路径中分离，形成可持久化的任务运行边界。
2. 补齐任务幂等、领取、超时、失败恢复、重复执行防护和死任务处理。
3. 在具备持久化协调后，再评估多实例、滚动发布和水平扩容；禁止仅增加 worker 数量伪装为高可用。
4. 提供基础运行状态、任务时间线、异常关联标识、监控和告警。
5. 补齐客户运营的最小闭环：邀请、找回密码、离职停用、待审批提醒和管理员基础自助处理。

### 完成标准

- API 重启、任务失败和长任务中断有明确且可验证的恢复语义。
- 客户和支持人员能够定位任务卡在哪一步，并知道下一步恢复动作。
- 系统不依赖原开发者人工盯守才能完成日常运行。
- 在 R3 验收前，不对外承诺多实例高可用或企业级 SLA。

### 12.2 当前 R2 实施记录

- 2026-07-12：发现并修复真实模型验收的假阳性：此前 `--use-model` 允许部门草案在模型失败后自动降级为模板成果，部署 smoke 不会断言 `generated_by == model`，因此“模型 smoke 通过”可能仅代表 fallback 通过。该问题已记录为 R2.3-R2.4 的实现证据，但 R2 不因此完成。
- 2026-07-12：`ai_artifacts` 新增并持久化 `generation_source` 与 `generation_fallback_reason`。新生成的部门成果会明确记录 `model`、`fallback` 或历史/人工默认来源；调度事件、命令读模型、团队工作台和销售报价审批返回统一暴露相同来源字段，避免同一成果在不同客户界面呈现出不一致事实。
- 2026-07-12：桌面端研发工程部、财务法务风控部成果卡及员工生成反馈会明确显示“真实模型生成”或“模板兜底生成”；出现兜底时显示机器可读的失败类别，不能再把模板成果表述为模型成果。
- 2026-07-12：部署 smoke 与主链 smoke 增加 `--require-real-model`。该模式要求显式上游 Base URL、模型名称和通过环境变量注入的 API Key；任一产品、运营、风控部门草案不是 `model` 即失败。默认本地 smoke 保持 fallback 验证范围，不会被误称为真实模型 E2E。
- 2026-07-12：已验证模型成功与失败语义：新增 4 个来源契约测试通过；本地 OpenAI-compatible 上游契约演练确认三次真实请求均进入上游，产品、运营、风控三份草案均为 `model`；另一次故意返回失败的演练被门禁以非模型来源拒绝。完整后端回归为 `281 passed, 2 skipped`；桌面端 `npm run build` 和 `npm run test:phase1-ui` 通过。
- 未关闭项：GitHub Actions 中的 PostgreSQL workflow 仍待网络恢复后推送并实际执行；真实模型门禁尚未绑定到受管 CI 密钥/发布环境；超时、限流、格式异常和恢复的外部模型提供商契约仍需扩充；隔离临时环境中的“桌面安装包 -> 后端 -> PostgreSQL -> 真实模型”全链 E2E 和旧版本升级演练尚未完成。因此 R2 仍不得勾选完成。
